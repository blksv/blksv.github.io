Objects and ORMs

Published: 2022-11-12

Up:

    Notes on programming

Somehow, a long time ago, object-oriented design practices went down a crooked path of equating domain objects with software objects. While justified in some use cases, such confusion is disastrous when applied as a general principle.

Why is the distinction between domain and software object important? Because software operates on a different conceptual level. Try to imagine a manager being sent to a DatabaseConnection in the real world. Nonsense.

An object in a program is an actor with some behavior and state. It's not what we usually have when representing domain objects in our programs. In most cases, we have some data related to the domain objects and rules of operations with this data. And trying to make datasets actors leads to very awkward designs (when you question yourself in which class you should put a method, say, should it be Manager.fire(Employee) or Employee.fire() you're doing exactly this).

But this dubious activity is the main purpose of ORMs; let alone the much more dubious case when ORM is used as a glorified and terribly inefficient query builder just proxying data requests to the DB.

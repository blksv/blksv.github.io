GraphQL

Published: 2022-11-08

Up:

    Notes on programming

I'm strongly against it as I see only one very narrow use case for that technology (see below), which on its own is somewhat doubtful.

The problems are:

  1. You usually have to maintain a consistent description of your data scheme at the following levels:

    • the data storage itself,
    • data storage ↔ ORM connector,
    • ORM connector ↔ GraphQL resolver,
    • GraphQL queries at the consumer.

    This. Is. Hard.

  2. To make the «ORM connector ↔ GraphQL resolver» subsystem use the database efficiently is on the brink of impossible. In fact, it can bring the DB down through the N+1 problem even on the simplest queries. Trying to fix the problem with the extensive caching creates a whole host of very intricate cache inconsistency bugs.

  3. Exposing GraphQL APIs is about the same as exposing direct SQL to your database. You really don't want this machine gun to get into the wrong hands. Frontend developers' hands are usually wrong too.

  4. Being RPC-styled, GraphQL requests cannot be rerouted at the web server / load balancer level because you need to parse the query to determine where it should go. This creates unnecessary infrastructure overhead when you decide to spin off some parts of your application server as separate microservices.

The only case for GraphQL I can come by is when a lot of teams work on different backend-for-fronted services with so different and so fluent data requirements, that it is more practical to have the «data-providing» team work on a generic GraphQL interface instead of implementing requests from «data-consuming» teams.
